## 2. MCP vs Skills — The Great Agent Integration Debate (391 HN points)

**Source:** [david.coffee](https://david.coffee/i-still-prefer-mcp-over-skills/), [HN #47712718](https://news.ycombinator.com/item?id=47712718)

A sharp piece arguing MCP is architecturally superior to Skills for AI agent integrations. 323 comments. Key arguments:

**Why MCP wins for service integration:**
- API abstraction: LLM calls `devonthink.do_x()`, MCP server handles the rest — no CLI required
- Zero-install remote usage: remote MCP servers work from any client without local setup
- Seamless auto-updates: server updates propagate to all clients instantly
- OAuth auth: cleaner than dumping secrets in .env files
- True portability: works from Mac, phone, web — cross-client
- Sandboxing: controlled interface vs. raw execution power
- Smart discovery: modern clients load tools only when needed (Tool_search)

**Why Skills fail for service integration:**
- CLI dependency: ChatGPT, Perplexity, standard Claude can't run CLIs — dead on arrival for most users
- Deployment mess: binaries, NPM, uv — fragmented installation chains
- Secret management nightmare: ephemeral environments wipe secrets
- Context bloat: loading entire SKILL.md vs. single tool signature
- Ecosystem fragmentation: some tools support `npx skills`, others don't; no universal standard

**HN community consensus:**
- MCP's main value is its **uptake and standardization** — it's "for agents" in a way CLIs never were
- Tool_search largely solved MCP discoverability issues
- Best practice: build the MCP server properly (optimized prompts), let the agent use it; don't hand-roll descriptions in skill files
- For pure knowledge tasks, Skills work fine — for persistent service access, MCP is strictly better

**What this means for Alizé:**
- Alizé agents should expose MCP-compliant connectors for enterprise tools (CRM, ERP, HR systems)
- Alizé's "managed service" positioning means it handles MCP server maintenance — customers get plug-and-play integrations without CLI hell
- This is a strong product differentiator: "We maintain the MCP layer so you don't have to"
- Skills-only approach locks out browser/chat-based AI usage — MCP unlocks all clients

## 3. Pipe-Connected AI Agents — "Incredible Machine" Pattern Goes Mainstream

**Source:** HN comment on #47719486 thread

An HN commenter (vote score suggesting broad agreement) described a vision of **pipe-connected AI agents**:

> A Firefox agent downloading pictures of cats → piping to a graphics program drawing mustaches → piping to a moviemaker → piping to a video uploader — all assembled by a non-coding user in a visual "Incredible Machine"-like explorer.

This is a compelling UX metaphor for agent workflow composition. The pipe-and-filter model (Unix philosophy) applied to AI agents:

- Each agent is a single-purpose tool with defined inputs/outputs
- Users compose workflows visually by connecting outputs to inputs
- No coding required — drag-and-drop assembly
- The "glue" is the pipe, not prompts

**What this means for Alizé:**
- Pipe-connected workflows could be Alizé's **user-facing product concept**: "Build your agent workflow like plumbing"
- French/Belgian/Swiss enterprises that don't want to code prompts could use a visual pipe builder
- This aligns with low-code/no-code trends in European SME software
- Competitor angle: most US AI agent startups are prompt-first; a pipe-first UX would be genuinely differentiated

## Product Direction Recommendations

1. **MCP-first integration strategy**: Build Alizé's connectors as MCP servers, not CLIs or skills. This ensures compatibility across all AI clients (Claude, ChatGPT, Perplexity, etc.) and future-proofs the integration layer.

2. **Pipe-connected workflow UX**: Explore a visual workflow builder where non-technical users can connect agent tasks (file processor → summarizer → CRM updater → email sender). Differentiates from prompt-only competitors.

3. **Sovereign stack positioning**: Explicitly market Alizé as running on EU-hosted infra (Mistral, sovereign clouds). The France Linux migration creates a halo effect — government buyers will prefer vendors with EU data residency.

4. **Government vertical**: DINUM migration creates a wedge into French public sector. First mover advantage for an AI agent provider that can sign EU data processing agreements and run on French sovereign cloud.
