## Signal 1: MCP Ecosystem Explodes — 17,000 Servers in One Week

The most significant infrastructure signal this cycle: MCP (Model Context Protocol), Anthropic's open protocol for connecting AI agents to external tools and data sources, has reached an inflection point. MCPpedia reported 17,000 MCP servers published in a single week — a growth rate that suggests the protocol has won the integration standard war.

MCP's trajectory mirrors the early days of REST APIs: a dominant player (Anthropic/Anthropic Claude) releases an open protocol, and the ecosystem rapidly builds around it. The key difference from previous integration attempts (LangChain,LlamaIndex, etc.): MCP is vendor-neutral enough that OpenAI, Google, and local models can implement it, while being Anthropic-sponsored enough to maintain coherence.

The competitive implication is significant. Once 17,000 MCP servers exist, switching away from MCP becomes costly for any AI vendor — the network effects of existing integrations create lock-in comparable to REST in the web era.

**What this means for Alizé:**

MCP is infrastructure, not product. The opportunity is not in building MCP servers — it's in being the operator that connects MCP servers to enterprise workflows.

Alizé's managed AI service should be MCP-first: "We connect your Claude agents to your existing tools via MCP — your CRM, your ERP, your internal APIs — and we own the integration, the monitoring, and the accountability." This is a more concrete and defensible positioning than "we deploy AI agents."

The MCP ecosystem also creates a clear product expansion path: Alizé could maintain a curated registry of enterprise-grade MCP servers for common business functions (SAP, Salesforce, Microsoft 365), positioned as a "certified MCP integration partner." This mirrors how Red Hat built value on top of Linux by certifying enterprise Linux configurations.

## Signal 2: Freestyle — Full Linux Sandboxes for AI Coding Agents

A Launch HN story (Freestyle, freestyle.sh) dropped this cycle with a compelling infrastructure pitch: building cloud sandboxes that are interchangeable with EC2 from an AI agent's perspective, with capabilities that standard cloud VMs can't match:

- **Memory forking**: Fork an entire running VM's memory state in under 400ms. Animations, browser sessions, running processes — all preserved across forks. This is not filesystem snapshotting; it's full memory state transfer.
- **500ms cold start**: VMs boot in half a second
- **Full Linux + hardware virtualization**: eBPF, FUSE, systemd init — whatever the AI expects from Debian, it gets
- **Own bare metal**: They found cloud bare metal pricing equivalent to total hardware cost, so they bought their own racks

The founding thesis: the first generation of AI coding agents had minimal tools. The current generation has full computer access. The next generation needs infrastructure that can match — instant cloning of running environments, sub-second provisioning, hardware-level isolation.

**What this means for Alizé:**

This is enterprise-grade AI infrastructure, not consumer tooling. Freestyle's pitch — "replicate the human dev loop at AI scale" — is exactly the infrastructure problem that large enterprises deploying AI agents will face.

For Alizé, the insight is architectural: **AI agent infrastructure is becoming a distinct category**. The era of "AI runs on your existing cloud" is ending. AI agents need specialized compute environments (isolation, state forking, fast provisioning) that general-purpose cloud was not designed for.

This creates an opportunity for a new kind of managed service: not just "we deploy your AI agents" but "we run your AI agents on infrastructure designed for AI agents." Alizé should investigate whether this is a differentiating capability or a commodity quickly captured by hyperscalers.

The HN response was moderate (56 points, 1 hour old at time of research), but the technical approach is sound and the timing is right.

## Signal 3: Reducto's Deep Extract — Agent Harness for Document Intelligence

Reducto launched "Deep Extract," a new approach to structured document extraction that uses an agent harness architecture: the extraction agent verifies and corrects its own output through an autonomous feedback loop, rather than relying on a single-pass model inference.

Key details:
- **Agent-in-the-loop verification**: Unlike traditional single-pass OCR + LLM extraction, Deep Extract breaks extraction into steps, verifies intermediate outputs, and iterates until results are accurate
- **99-100% field accuracy** on long, complex documents — even outperforming expert human labelers
- **Scale**: 28+ million fields extracted on documents up to 2,500 pages in production beta
- **Target use cases**: Invoice line items, brokerage statements, equipment manifests, regulatory filings

The core problem being solved: models get lazy on long, repetitive tasks. Given 1,000 invoice line items to extract, a single-pass model will often stop short, consolidate entries, or skip rows. The agent harness approach forces thoroughness through verification cycles.

**What this means for Alizé:**

Reducto's Deep Extract is a leading indicator of where high-stakes document processing is going: not prompt engineering, but agentic pipelines with built-in verification. The "agent harness" pattern — autonomous verification + self-correction — is the architectural answer to the problem of AI unreliability that every enterprise eventually hits.

For Alizé's managed services, this pattern should be standard for any repetitive, high-volume document workflow:
- Invoice processing → extraction → cross-check totals → flag discrepancies → human review for exceptions
- Customer email routing → classification → confidence scoring → escalate low-confidence to human
- Contract review → clause extraction → compliance check → flag problematic clauses

The principle: **every AI workflow that processes important data should have a self-verification step before the output is trusted**. This is the difference between "AI that works" and "AI you can bet your business on."

## Signal 4: Claude Code Feb Updates Break Complex Engineering — 318 HN Points

A GitHub issue titled "Claude Code is unusable for complex engineering tasks with the Feb updates" reached 318 points on HN this cycle, with a detailed technical analysis and a response from Anthropic's Boris on the Claude Code team.

The issue: two changes in February materially degraded Claude Code's performance on complex, long-horizon engineering tasks:
1. **Adaptive thinking replaces fixed budgets** (Feb 9, Opus 4.6 launch): The model now decides how long to think, rather than using a fixed budget. The model tends to think less than a fixed budget would have allocated for complex tasks.
2. **Medium effort=85 default** (Mar 3): Anthropic found effort=85 was a better intelligence/latency/cost balance for most users, so they changed the default with a dialog notification.

The result: developers doing complex engineering (large refactors, multi-file changes, deep debugging) found Claude Code was producing significantly less thorough work. The thinking was shorter, the work was shallower.

**What this means for Alizé:**

This incident is a case study in why managed AI services matter more than raw model capability.

Anthropic — the premium enterprise AI vendor — changed defaults that materially affected output quality, and the change was subtle enough that most users wouldn't notice until something went wrong. The fix is opt-out (`CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING`, `showThinkingSummaries: true`), but that requires knowing something broke and how to fix it.

This is the AI reliability problem in microcosm: even tools from the most enterprise-focused AI company are still iteratively breaking and improving in ways that affect output quality. Enterprises using AI without oversight don't know when a model update degrades their workflows.

Alizé's managed service model — where Alizé owns the AI operation and monitors output quality — is the answer. When Anthropic pushes a model update, Alizé notices. When Claude Code's behavior changes, Alizé catches it. The enterprise customer doesn't need to debug their AI vendor.
