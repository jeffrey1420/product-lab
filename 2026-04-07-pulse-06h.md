## Signal 3: Calx — Behavioral Corrections as Enforceable Rules (MCP Server)

A new tool that directly addresses the AI agent reliability problem: Calx (github.com/getcalx, v0.5.0, MIT license) is an MCP server that tracks behavioral corrections humans make to AI agents and promotes recurring corrections to enforced rules.

The empirical basis is strong: the developer built a production system with 6 AI agents across 82,000 lines of code in 20 days for $250, keeping obsessive correction logs. When transferring 237 corrections as rules to a new agent, the new agent made 44 new mistakes — 13 in categories the rules explicitly covered. Rules were present in context. The behavior wasn't.

Meta's Superintelligence Labs published HyperAgents (arXiv:2603.19461, March 2026) the same week, finding the complementary result: improvements transfer across domains when embodied in executable mechanisms (persistent memory, performance tracking, eval loops) — not when written as rule text. Two independent studies, same conclusion.

Calx's architecture:
- Captures corrections via CLI or MCP protocol
- Detects recurrence via Jaccard keyword similarity, auto-promotes at 3x threshold
- Promotes to enforced rules and hooks injected at session start
- Scopes rules per domain/directory
- Handles token discipline (prevents context compaction from destroying correction signal)
- Runs as FastMCP server over Streamable HTTP with SQLite locally

**What this means for Alizé:**

Calx is a leading indicator of where the AI agent reliability tooling market is heading: from information retrieval (vector databases, memory systems) to behavioral enforcement (correction tracking, eval loops, enforced rules). Alizé should evaluate Calx as a potential integration for its managed service — if Alizé's agents can benefit from Calx's correction-tracking layer, it strengthens the reliability story.

The broader insight: the market is beginning to distinguish between "AI that knows things" and "AI that behaves correctly." Alizé's governance model is already positioned on the "behaves correctly" side. Calx is a concrete technical implementation of this principle.

## Signal 4: Capability Injection — Enterprise Agent Authorization Pattern

A detailed HN discussion (from an architect who has been building production agent tooling) proposed "Capability Injection (CI)" as the enterprise-grade pattern for AI agent authorization — drawing a direct analogy to Dependency Injection in software architecture.

The core argument: AI agents currently operate like pre-DI software — with static permissions, hardcoded tools, or broad service accounts. This works for demos but breaks in enterprise settings where governance and auditability are non-negotiable.

The CI pattern:
- Agents declare the capabilities they need for a specific purpose (not possessed permissions)
- A governing layer (realm) evaluates the request, resolves the human principal, checks policy
- Injects a scoped, time-bound delegation token
- Agent runs strictly within that scope and never holds standing privileges
- Grounded in human identity — integrated with LDAP or Active Directory
- Every injected capability traceable to a person, for a purpose, for a duration

This is structurally what Alizé is building operationally. The pattern validates the approach but shows the technical implementation gap: Alizé currently provides the governance process; the technical implementation (scoped tokens, AD integration, policy evaluation) would be the product layer on top.

**What this means for Alizé:**

CI is an architectural pattern, not a product. But it points to what a mature AI governance product looks like: policy-driven capability injection with enterprise identity integration. Alizé's current positioning — human accountable operator, audit logs, scope constraints — is the market education phase. The eventual product is a technical governance layer that enforces these constraints automatically.

The practical near-term implication: Alizé should be able to articulate its governance model in terms of CI when talking to technical decision-makers. "We're the managed service that implements the CI pattern for you" is a clearer pitch than "we do AI governance."
