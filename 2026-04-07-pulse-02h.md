## Signal 2: Post-Quantum Cryptography — 2029 Is the Hard Deadline

A quietly urgent story breaking on HN this cycle (Filippo Valsorda, Google cryptography engineer): the timeline for cryptographically-relevant quantum computers (CRQC) has compressed dramatically. Two papers in the past week fundamentally changed the risk picture:

1. **Google revised down** the estimated number of logical qubits and gates required to break 256-bit elliptic curves (NIST P-256, secp256k1), making attacks on TLS MitM feasible with minutes of compute on superconducting qubit architectures
2. **Oratomic showed** 256-bit curves can be broken with as few as 10,000 physical qubits using neutral atom architectures with non-local connectivity

Google's Heather Adkins and Sophie Schmieg published an explicit migration timeline: **2029 is the deadline**. Scott Aaronson's public warning frames it as analogous to nuclear fission research going dark between 1939-1940 — the window for action is narrower than the expert community expected.

The critical distinction for enterprise security teams:
- **Key exchange (ML-KEM/FIPS 203)**: Migrate now — adversaries can harvest encrypted traffic today and decrypt later ("harvest now, decrypt later")
- **Digital signatures/authentication**: Less urgent — forging signatures retroactively is only valuable if paper documents are fully replaced by digital equivalents

OpenSSH, OpenSSL, and most cryptographic libraries already support ML-KEM — deployment is technically feasible today. The bottleneck is enterprise change management.

**What this means for Alizé:**

This is a wedge opportunity for AI-powered enterprise operations. PQC migration is a large, messy, multi-team problem: auditing TLS configurations across thousands of endpoints, coordinating with hardware vendors, updating cryptographic libraries, managing hybrid deployments during transition. It's exactly the kind of distributed coordination problem that AI agents can accelerate — but it requires human oversight for the compliance and validation steps.

Alizé could position itself as the "AI operations partner for PQC migration" — using AI agents to audit cryptographic configurations, generate remediation scripts, track certificate rotation deadlines, and produce audit-ready compliance reports. The market is large and the deadline is hard (2029). Every French enterprise will face this.

More broadly: the 2029 deadline is forcing enterprises to treat cryptographic infrastructure as a board-level priority for the first time. This creates organizational air cover for AI governance initiatives — if the CISO is worried about quantum migration, the CISO is also worried about AI agent security, and Alizé's governance positioning becomes relevant in that conversation.

## Signal 4: Adobe Hosts File Modification — Enterprise Software Trust Erosion

OSNews reported (246 HN points) that Adobe Creative Cloud silently modifies the Windows and macOS hosts file to detect whether the app is already installed. When you visit adobe.com/home, JavaScript loads an image from detect-ccd.creativecloud.adobe.com/cc.png — and if the DNS entry exists in the hosts file (which Adobe added without asking), the connection succeeds, revealing your install status to Adobe.

The reason for the hosts file modification: Adobe previously used localhost:ports to detect CC installation, but Chrome blocked localhost network access in 2024, so they switched to hosts file manipulation as a workaround. The thread title asks: "At what point does a commercial software suite become malware?"

**What this means for Alizé:**

The Adobe story is a small but vivid example of a broader pattern relevant to AI agent deployments: software with legitimate purposes expanding its access beyond what users expect or explicitly consent to. The AI agent parallel is obvious — agents that have broad system access and expand their scope in ways that are difficult to audit.

This matters for enterprise AI governance because it illustrates a point that's hard to explain abstractly: even trusted commercial software will opportunistically expand its access when blocked. AI agents, which are designed to pursue goals flexibly, are inherently more likely to exhibit this behavior than traditional software.

Alizé's governance model — scope constraints, audit logging, human escalation for out-of-scope actions — is the structural answer to this problem. The pitch isn't "our AI won't do this" — it's "our AI operates within constraints that are technically enforced and visibly logged." Adobe's behavior is a reminder that the trust-but-verify model is necessary even for established commercial vendors, not just AI startups.

## Signal 5: Hippo Memory — Cross-Tool Agent Memory Becomes a Product

Hippo Memory (47 points, kitfunso/hippo-memory) launched this cycle: a biologically-inspired memory system for AI coding agents with three core mechanisms — decay, retrieval strengthening, and consolidation. Unlike vector databases that just save everything and search later, Hippo implements what the author calls "knowing what to forget."

Key features:
- **Decay**: Lower-confidence memories fade over time unless retrieval reinforces them
- **Cross-tool portability**: Works with Claude Code, Codex, Cursor, OpenClaw; memories stored as markdown files in the repo so they travel between tools
- **Session handoffs**: Explicit start/end lifecycle, successor sessions can resume without reading full transcripts
- **SQLite backbone**: Zero runtime dependencies, Node.js 22.5+, optional embeddings via @xenova/transformers
- **Explainable recall**: `--why` flag shows which terms matched, whether BM25 or embedding contributed, and source bucket

The framing is precise: "It's not a filing cabinet, it's a brain." The core insight — that AI memory shouldn't be infinite, and that forgetting is a feature not a bug — mirrors how human memory works and is starting to influence AI system design.

**What this means for Alizé:**

Hippo's approach validates the architectural direction Alizé should take for agent memory management: not "save everything" but "prioritize what matters, let noise decay." This is particularly relevant for long-running enterprise workflows where context windows are expensive and noisy.

More importantly, Hippo is a concrete implementation of the "agent memory" pattern that could eventually become a product component: an Alizé agent memory layer that works across client AI tools, retains high-value workflow patterns, and allows agents to hand off to each other with shared context. If Alizé's managed service runs agents across multiple clients, a shared memory layer with decay mechanics becomes a competitive moat — it gets smarter about common enterprise workflows over time.
