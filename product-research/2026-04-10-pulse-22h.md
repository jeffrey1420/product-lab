## 2. Samelogic — "The Missing Context Layer" Frames the Core Architecture Problem

**Source:** [Samelogic Blog](https://samelogic.com/blog/the-ai-agent-enterprise-revolution-building-the-missing-context-layer)

Samelogic published a sharp analysis of why enterprise AI agents fail in practice. The thesis: enterprises don't have a single data environment — they have a fragmented constellation of CRM, docs, HR, IT systems. AI agents need more than API access; they need **structured context** in each system to function reliably.

Four things AI-ready environments must provide:
1. **Structural context** for AI understanding (not just raw data)
2. **Reliable anchors** for consistent interaction (stable interaction points as interfaces evolve)
3. **Clear pathways** between systems (how to get from A to B)
4. **Permission frameworks** that translate to AI capabilities (RBAC → AI capabilities)

**What this means for Alizé:**
- This directly validates Alizé's "managed IA" positioning: someone has to build and maintain the context layer — that's the service
- The "context anchor" concept maps to Alizé's workflow design: each workflow step is a context anchor
- Pipe-connected agents (see pulse-18h) are literally the pathway-building metaphor
- Alizé should position itself as the "context layer company" for European PME — not another AI tool, but the integration layer that makes other tools work together

## 4. OWASP MCP Top 10 — Security Framework for AI Agent Tool Integration

**Source:** [OWASP MCP Top 10](https://mcpblog.dev/blog/2026-03-15-owasp-mcp-top-10)

The OWASP community published an MCP security framework. This is significant because:

## 5. Per-Tool Sandboxing for AI Agents — Granular Isolation Is the Future

**Source:** [MultiKernel](https://multikernel.io/2026/03/25/sandlock-mcp-per-tool-sandboxing/)

MultiKernel published **SandLock MCP** — per-tool sandboxing for AI agents. Instead of one sandbox for the entire agent, each MCP tool gets its own isolated sandbox:

- If one tool is compromised, the breach is contained to that tool's sandbox
- Each tool can have its own permission scope (file access, network, etc.)
- More granular than "all or nothing" sandboxing approaches

**What this means for Alizé:**
- Per-tool sandboxing addresses a real enterprise security concern: "I don't trust all my MCP tools equally"
- Alizé can differentiate on **security granularity** — enterprise buyers will pay a premium for sandboxed agent execution
- This aligns with the EU AI Act's requirements for measurable safety guarantees
- A potential Alizé feature: "Tiered trust levels per integration" — CRM gets full access, experimental MCP gets sandboxed
