## Signal 1: Anthropic's "Building Effective Agents" — The Industry's Definitive Reference

Anthropic published what is effectively the canonical taxonomy of agentic systems. This is the most practically relevant document for Alize in months.

**The core distinction Anthropic draws:**
- **Workflows**: LLMs and tools orchestrated through predefined code paths (predictable, consistent, lower latency)
- **Agents**: LLMs dynamically direct their own processes and tool usage (flexible, model-driven, higher latency/cost)

**Key practical recommendation:** "Find the simplest solution possible, and only increase complexity when needed. This might mean not building agentic systems at all."

**The five workflow patterns:**
1. Prompt chaining — sequential decomposition
2. Routing — classification to specialized handlers
3. Parallelization — sectioning + voting for speed/diversity
4. Orchestrator-workers — dynamic task distribution
5. Evaluator-optimizer — iterative refinement loops

**Augmented LLM** (the building block): retrieval + tools + memory. Anthropic specifically calls out their **Model Context Protocol (MCP)** as the integration layer for third-party tools — this is Anthropic's standardization play.

**Frameworks warning:** Anthropic recommends starting with direct LLM API calls — frameworks add abstraction layers that obscure prompts and responses. This is a direct challenge to the LangChain/AutoGen ecosystem narrative.

**The "when not to use agents" call:** Anthropic explicitly says that optimizing single LLM calls with retrieval and in-context examples is usually enough. This validates a key Alize positioning question: are French PME really at the "agent" stage, or are they still at the "workflow" stage? Answer: most are at workflow stage. Alize should sell workflows first, agents when justified.

**Alize angle:** This document essentially writes Alize's technical positioning. We build **structured workflows with human oversight** — not autonomous free-running agents. The language of "gates," "routing," "parallelization" maps directly to Alize's managed service playbook.

## Signal 3: Google Chrome Skills — Prompts as One-Click Tools

Google launched "Skills in Chrome" (April 14) — a feature that lets users save AI prompts as reusable one-click tools, accessible via "/" command or "+" button in Gemini in Chrome.

**What it does:**
- Save any Gemini prompt as a named "Skill"
- Replay it on any tab/page with one click
- Includes a library of pre-built skills (recipe ingredient substitution, gift comparison, document scanning)
- Skills work across all signed-in Chrome desktop devices

**Why this matters for Alize:**
- This is Google's answer to "prompt as tool" — the same concept as AI agent tool calling, but consumer-grade

- Google is rapidly commoditizing the "turn your best workflow into a reusable tool" pattern. Alize needs to differentiate on vertical depth (French business context) and governance, not on the pattern itself

## Signal 4: Multi-Agent Software Dev = Distributed Systems Problem

A well-trafficked HN discussion (April 14-15, ~23 hours old) on treating multi-agent pipelines as distributed systems with explicit verification gates.

**Key insight from practitioners:** The most reliable multi-agent pipelines use deterministic checks (compile, lint, tests) as hard floors of guarantee, with agentic reviewers providing soft probabilistic assertions. Each boundary between stages is a gate where misinterpretations get converted to detectable failures.

**The 20% task failure rate assumption:** One practitioner writes that their framework assumes ~20% task failure rate per agent step and designs for simple wipe-and-repeat of failed tasks. This is an important design constraint: don't try to make agents reliable at every step; make the overall protocol reliable despite individual failures.

**Routing patterns in practice:** The discussion highlights routing as a key pattern — easy/common questions to smaller models (Haiku), hard questions to larger models (Opus). Multi-model routing within a single workflow is now production practice, not theory.

**Alize angle:** These patterns directly map to Alize's managed service architecture. The "gate" concept is Alize's governance check. The 20% failure assumption justifies human oversight — the agent proposes, the human approves. The routing pattern justifies Alize's multi-model approach (Mistral for French language tasks, Claude for complex reasoning).

## Signal 6: AgentFM — P2P GPU Grid for AI Agents

AgentFM appeared on HN (9 points, 9 hours ago) — a single Go binary that turns idle GPUs into a P2P AI grid. This is an infrastructure-level signal worth watching.

**What it is:** Distributed compute for AI inference using idle consumer/enterprise GPUs, creating a peer-to-peer grid. Similar to how Folding@Home worked for protein folding.

**Why it matters for Alize:**
- If P2P compute becomes viable, it could disrupt the centralized cloud GPU economics that Alize relies on
- Early indicator of infrastructure commoditization below the model layer
- Not immediately relevant (Alize is a managed service, not an infra play) but worth monitoring
